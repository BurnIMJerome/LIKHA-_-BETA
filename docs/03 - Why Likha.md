# Why Likha

Likha exists for teams that want practical RPA and agentic AI automation without being forced into a large platform commitment on day one.

## Problems Likha Targets

- RPA platforms can become expensive before the first bot is in production.
- Small teams often need a focused desktop/browser automation tool before they need a full enterprise program.
- AI automation should work with the provider the team already uses.
- AI agents need governed access to reliable automation tools, machines, data, and human approval—not an unrestricted chatbot connected to production systems.
- Some organizations need on-premise or private infrastructure instead of mandatory cloud orchestration.
- Developers need clear activity behavior, readable workflow data, and direct control over runtime setup.

## Likha's Approach

- Build locally first.
- Run interactively in the designer when developing.
- Queue only when running unattended or through schedules/triggers.
- Use Control Room when coordination, queues, logs, schedules, and robot visibility matter.
- Let users bring their own AI settings.
- Let agents reason over changing work while reusable Likha flows perform controlled actions.
- Keep every agent bounded by its approved tools, runtime limits, credentials, and escalation rules.

## What Likha Is Not Trying To Be

Likha is not only an RPA recorder or a chatbot. It combines deterministic automation with agentic AI so a goal-driven agent can choose and coordinate approved tools while Likha keeps execution observable and controlled.
